## 今日研究方向建议

1. **跟进 TensorRT-LLM 中 KVCacheManagerV2 的 bug 修复与多模态 KV cache 块重用修复**：今日 [PR #12306](https://github.com/NVIDIA/TensorRT-LLM/pull/12306) 修复了 KVCacheManagerV2 的多个 bug（页损坏、max_blocks_per_seq 计算不足等），[PR #12472](https://github.com/NVIDIA/TensorRT-LLM/pull/12472) 则专门修复了分离式服务下多模态 KV cache 块的重用问题。这两项直接关联你的“KV Cache 管理”和“disaggregated prefill/decode”方向。建议你阅读 [PR #12306](https://github.com/NVIDIA/TensorRT-LLM/pull/12306) 的代码变更，重点关注 `max_blocks_per_seq` 计算逻辑的修正；然后结合 [PR #12472](https://github.com/NVIDIA/TensorRT-LLM/pull/12472) 中缓存键生成的改进，在本地 TensorRT-LLM 环境中，使用一个多模态模型（如 LLaVA）在分离式服务模式下验证 KV cache 块重用的正确性，并与修复前版本对比内存占用。

2. **研究 SGLang 对 DeepSeek V4 的支持与 cookbook**：今日 SGLang 有多个关于 DeepSeek V4 的文档更新（[PR #23617](https://github.com/sgl-project/sglang/pull/23617)）和 cookbook 添加（[PR #23605](https://github.com/sgl-project/sglang/pull/23605)）。这与你关注的“MoE dispatch”和“推理系统”方向相关。建议你阅读 DeepSeek V4 cookbook，重点关注 SGLang 如何配置和优化该 MoE 模型的推理路径（如 EP 调度策略）。你可以尝试在本地 SGLang 环境中，按照 cookbook 的指引部署 DeepSeek V4，并对比其与之前版本（如 V3）在吞吐量和显存占用上的差异，评估 SGLang 对新 MoE 架构的适配效率。

3. **分析 TensorRT-LLM 中 AutoDeploy 的 QKV 与 RoPE 融合优化**：今日 [PR #12357](https://github.com/NVIDIA/TensorRT-LLM/pull/12357) 在 AutoDeploy 中启用了 QKV 和 RoPE 融合，使 Llama 3.1 8B FP8 吞吐量提升 2.7%。这直接关联你的“性能优化”和“CUDA kernel 开发”方向。建议你阅读该 PR 的代码变更，重点关注融合 kernel 的实现细节（如如何将 RoPE 计算嵌入 attention 流程）。你可以尝试在本地 TensorRT-LLM 环境中，使用 Llama 3.1 8B FP8 模型复现此优化，对比开启/关闭融合后的吞吐量，并分析其与手动编写融合 kernel 的性能差距。

4. **评估 vLLM 中 `chat_template_kwargs` 修复对多轮对话的影响**：今日 [PR #40713](https://github.com/vllm-project/vllm/pull/40713) 修复了 `HYV3ReasoningParser` 初始化时可能修改共享 `chat_template_kwargs` 对象的 bug。这与你“近期关注”的“节约 tokens 的多轮对话方案”相关。建议你阅读该 PR 的代码变更，理解其如何避免对共享对象的副作用。你可以尝试在本地 vLLM 环境中，模拟多轮对话场景（如连续发送多个请求），验证修复前后 `chat_template_kwargs` 的行为差异，并思考此模式对你正在优化的 prompt 管理方案有何启示。
