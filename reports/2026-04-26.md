## 今日研究方向建议

1. **跟进 vLLM 在 P/D 分离场景下的完整 KV 块卸载策略**：今日 [[PR #40346](https://github.com/vllm-project/vllm/pull/40346)](https://github.com/vllm-project/vllm/pull/40346) 实现了预填充实例将所有 KV 块卸载到 CPU（而非仅增量），以确保远程解码节点能获取完整缓存。这与你关注的“disaggregated prefill/decode”和“KV Cache 管理”直接相关。建议你阅读该 PR 的 `OffloadingConnector` 代码变更，重点关注 `start_block_idx` 重置为 0 的逻辑。然后，在本地 vLLM 环境中，使用一个长序列模型（如 LLaMA-3-8B）复现 P/D 分离场景，对比修复前后解码节点的缓存命中率与端到端延迟。

2. **研究 SGLang 中废弃 Triton 内核并合并 MoE 激活函数的优化**：今日 [[PR #23707](https://github.com/sgl-project/sglang/pull/23707)](https://github.com/sgl-project/sglang/pull/23707) 废弃了重复的 `act_and_mul_triton` 内核，将 `filter_expert` 功能合并到更快的 JIT CUDA `silu/gelu_and_mul` 内核中，减少约 100 行代码。这与你“MoE dispatch”和“CUDA kernel 开发”方向高度相关。建议你阅读该 PR 的代码变更，重点关注 JIT 内核如何通过编译时模板参数实现零开销的 `filter_expert` 集成。然后，尝试在 SGLang 中，对一个 MoE 模型（如 Mixtral 8x7B）对比优化前后的解码吞吐，并分析 Triton 与 JIT CUDA 内核在 MoE 场景下的性能差异。

3. **评估 TensorRT-LLM 中扩散模型请求重构对推理流程的影响**：今日 [[PR #13313](https://github.com/NVIDIA/TensorRT-LLM/pull/13313)](https://github.com/NVIDIA/TensorRT-LLM/pull/13313) 将 `VisualGenParams` 嵌入 `DiffusionRequest` 并简化 `generate()` 输入。这与你“多模态模型的量化/低精度推理优化”方向相关。建议你阅读该 PR 的接口变更，重点关注 `generate()` 输入简化为 `Union[str, List[str]]` 后的调用链。然后，尝试在 TensorRT-LLM 中，使用一个扩散模型（如 Stable Diffusion）验证新接口的兼容性，并对比重构前后的请求处理延迟。

4. **分析 SGLang 在 CPU 上优化 Qwen3.5 模型的算子融合与 TP 填充**：今日 [[PR #19484](https://github.com/sgl-project/sglang/pull/19484)](https://github.com/sgl-project/sglang/pull/19484) 为 SGLang 添加了 Qwen3.5 在 CPU 上的优化，包括数据类型融合和 TP 填充。这与你“性能优化”和“推理系统”方向相关，尤其涉及跨平台部署。建议你阅读该 PR 的代码变更，重点关注其如何通过算子融合减少 CPU 上的内存带宽瓶颈。然后，尝试在 CPU 环境中，使用 SGLang 加载 Qwen3.5 模型，对比优化前后的推理速度与内存占用，并评估此优化思路是否可迁移到其他模型或 GPU 平台。
