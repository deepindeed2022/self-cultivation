## 今日研究方向建议

1. **跟进 CUTLASS FP8 GEMM 2xAcc 技术**：今日 NVIDIA/cutlass 的 [PR #3149](https://github.com/NVIDIA/cutlass/pull/3149) 在 Hopper CuTeDSL 中实现了 FP8 GEMM 的双累加器方案，防止精度损失。这与你关注的 FP8 量化推理直接相关。建议你阅读该 PR 的 kernel 实现细节，重点关注双累加器如何周期性地提升临时累加器到主累加器。然后，尝试将此技术思路与你熟悉的 TensorRT-LLM 或 vLLM 中的 FP8 GEMM 实现进行对比，评估其是否可被借鉴以改善现有推理引擎中 FP8 矩阵乘法的数值稳定性。

2. **研究 vLLM 多 KV 组卸载的存储扩展**：今日 vLLM 的 [PR #39403](https://github.com/vllm-project/vllm/pull/39403) 扩展了 KV 卸载连接器以支持多个 KV 组的存储操作。这直接关联你关注的“KV Cache 管理”和“分布式推理”方向。建议你阅读该 PR 的代码变更，重点关注 `kv_offload` 模块中如何设计多组存储的接口与数据结构。然后，在本地 vLLM 环境中，尝试使用一个需要多 KV 组存储的模型（如某些 MoE 或长上下文模型）来验证此功能，并与单组存储方案进行内存占用和吞吐的基准测试。

3. **分析 FlashInfer SM100 持久化 prefill 内核的死锁修复**：今日 FlashInfer 的 [PR #3155](https://github.com/flashinfer-ai/flashinfer/pull/3155) 修复了 SM100 持久化 prefill 内核中因子进程 CUDA 上下文未初始化导致的死锁问题，改用物理 SM 计数。这与你“性能优化”和“CUDA kernel 开发”方向高度相关。建议你仔细阅读该 PR 的修复逻辑，理解物理 SM 计数与逻辑 SM 计数在持久化内核中的区别。你可以将此修复作为案例，分析在你的 Triton 或 CUDA kernel 开发中，当涉及多进程或持久化内核时，可能遇到的类似上下文初始化问题，并提前设计防御性代码。

4. **评估 SGLang JIT 内核兼容 FA3 无 out 参数的修复**：今日 SGLang 的 [PR #23717](https://github.com/sgl-project/sglang/pull/23717) 使 JIT 内核兼容没有 out 参数的 FlashAttention v3 内核，避免参数不匹配错误。这关联你“推理系统”和“性能优化”方向。建议你阅读该 PR 的代码变更，理解其如何通过参数检测或默认值处理来兼容不同版本的 FA3 内核。然后，尝试在本地 SGLang 环境中，使用一个依赖 FA3 的模型来验证此修复，并思考这种兼容性设计是否值得借鉴到你正在参与或关注的推理引擎（如 vLLM 或 TensorRT-LLM）的注意力后端中。
