## 今日研究方向建议

1. **评估 SGLang 升级到 PyTorch 2.11.0 对推理性能与兼容性的影响**：今日 SGLang 的 [commit #21247](https://github.com/sgl-project/sglang/commit/...) 将 PyTorch 依赖升级到 2.11.0。这与你关注的“transformers 版本更新对 LLM 推理框架兼容性的影响”直接相关。建议你阅读该 PR 的代码变更，重点关注升级后对 CUDA graph 捕获、Eager mode 下的 kernel 选择以及连续批处理调度的影响。然后，在本地 SGLang 环境中，使用一个典型 LLM（如 LLaMA-3-8B），对比 PyTorch 2.10.0 与 2.11.0 在相同配置下的首 token 延迟、解码吞吐和显存占用，并记录是否有任何 kernel 报错或性能回退。

2. **跟进 TensorRT-LLM 中 nvfp4 权重动态更新的实现，评估其对 MoE 模型在线量化的价值**：今日 [NVIDIA/TensorRT-LLM [PR #12320](https://github.com/NVIDIA/TensorRT-LLM/pull/12320)](https://github.com/NVIDIA/TensorRT-LLM/pull/12320) 支持了 nvfp4 权重的动态更新，包括 MoE 的 FC2 输入缩放和动态量化。这与你关注的“量化（FP4）”、“MoE dispatch”和“Blackwell 上 FP4 推理”方向高度相关。建议你阅读该 PR 的代码变更，重点关注 `update_weight` 函数如何实现 nvfp4 权重的在线重量化，以及 MoE 专家权重的更新路径。然后，尝试在 TensorRT-LLM 中，使用一个 MoE 模型（如 Mixtral 8x7B），对比静态 nvfp4 量化与动态 nvfp4 权重更新在推理精度和吞吐上的差异，并分析动态更新对显存和延迟的开销。

3. **研究 TensorRT-LLM 中滑动窗口注意力（SWA）工作区的清理，评估其对长序列推理的优化**：今日 [NVIDIA/TensorRT-LLM [PR #13346](https://github.com/NVIDIA/TensorRT-LLM/pull/13346)](https://github.com/NVIDIA/TensorRT-LLM/pull/13346) 清理了 SWA 的旧有工作区，改用新的基数搜索树和占位符块机制。这与你关注的“KV Cache 管理”和“性能优化”方向直接相关。建议你阅读该 PR 的代码变更，重点关注新的基数搜索树如何管理 SWA 的 KV 块，以及占位符块如何减少内存碎片。然后，在 TensorRT-LLM 中，使用一个支持 SWA 的模型（如 Mistral-7B），对比清理前后在长序列（如 32K tokens）推理中的显存占用和首 token 延迟，并分析新机制对连续批处理中不同长度请求的调度效率。

4. **分析 Triton 中 warp specialization 死锁检测的修复，评估其对自定义 kernel 开发的影响**：今日 [triton-lang/triton [PR #10192](https://github.com/triton-lang/triton/pull/10192)](https://github.com/triton-lang/triton/pull/10192) 修复了 warp specialization 代码中的死锁检测，通过为每个 CTA 存储运行时掩码来替代静态分析。这与你“CUDA/Triton kernel 开发”的背景直接相关。建议你阅读该 PR 的代码变更，重点关注 `deadlock_detection` 函数如何从静态分析改为运行时掩码判断，以及这种改变如何避免因分区提前结束导致的误判。然后，尝试在你自己开发的或 SGLang/vLLM 中的 warp specialization kernel（如 MoE dispatch 或 attention kernel）中，应用该修复逻辑，并验证其对 kernel 稳定性和性能的影响。
