1.  **跟进 FlashInfer v0.6.9rc1 中的 FP4 矩阵乘法后端与 MoE routing_replay_out 支持**：今日 FlashInfer 发布了 v0.6.9rc1，新增了 SM120 上的 FP4 矩阵乘法后端，并支持 MoE 内核的 `routing_replay_out`。这直接关联你“量化（FP4）”和“MoE dispatch”方向。建议你阅读该 release 的变更日志，重点关注 FP4 后端的 kernel 实现细节（如是否依赖特定 SM 特性）。然后，在你的开发环境中编译该 rc 版本，并尝试将其集成到 vLLM 或 SGLang 中，针对一个 MoE 模型（如 Mixtral）验证 FP4 推理的正确性与性能，并与 FP8/W4A16 方案进行对比。

2.  **研究 NVIDIA Model-Optimizer 新增的 KV cache FP8/NVFP4 量化配方**：今日的 PR #1334 为 Model-Optimizer 添加了 `fp8_cast` 和 `nvfp4_cast` 两个 KV cache 量化配方，并让 `--recipe` 参数完全决定 KV cache 配置。这与你“KV Cache 管理”和“量化”方向高度相关。建议你仔细阅读该 PR 的 YAML 配方文件，理解其如何通过配置直接指定 KV cache 的量化格式。你可以尝试使用这些新配方，在你关注的模型（如 LLaMA 或 BERT）上，对比不同 KV cache 量化方案（FP8 vs NVFP4）对推理吞吐和精度的影响，并评估其与 TensorRT-LLM 或 vLLM 中现有 KV cache 量化机制的兼容性。

3.  **验证 Transformers v5.6.2 对 Qwen 3.5/3.6 MoE FP8 模型的修复**：今日 Transformers 发布了 v5.6.2 补丁，修复了 Qwen 3.5 和 3.6 MoE 模型在 FP8 下无法运行的问题。这与你“近期关注”的“transformers 版本更新对 LLM 推理框架兼容性的影响”直接相关。建议你立即升级到 v5.6.2，并尝试在 vLLM 或 SGLang 中加载一个 Qwen 3.6 MoE FP8 检查点，验证其是否能正常推理。同时，对比 PR #45610 中的修复细节，理解其解决的具体问题（如 `WeightConverter` 缺失），这有助于你未来快速定位类似兼容性问题。

4.  **分析 vLLM 中 MoE workspace 内存泄漏的修复**：今日的 PR #39226 修复了 vLLM 中 MoE workspace 调整大小时因未及时释放 GPU 内存导致的内存泄漏问题。这直接关联你“性能优化”和“MoE dispatch”方向。建议你阅读该 PR 的代码变更，重点关注 `empty_cache` 调用的添加位置和时机。你可以尝试在本地 vLLM 环境中，使用一个 MoE 模型（如 Mixtral 8x7B）并模拟动态调整 batch size 的场景，对比修复前后的 GPU 内存占用曲线，以量化此修复对高并发推理稳定性的提升。
